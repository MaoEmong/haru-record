# 위치 경로 정리 Deep Interview 결과

날짜: 2026-04-29
주제: GPS 튐으로 지저분해지는 지도 경로 표시 개선

## 문제 정의

현재 앱은 실제 위치 포인트를 수집하고 있지만, GPS가 순간적으로 튀는 경우 지도 경로에 도로/동선과 무관한 긴 선이 생긴다. 사용자는 원본 좌표 보존보다 화면에서 하루 이동을 이해하기 쉬운 경로 표시 품질을 원한다.

## 대상 사용자

- 앱을 실제로 들고 이동하며 하루 위치 기록을 확인하는 사용자
- 오늘 탭, 오늘 기록, 하루 상세 지도에서 이동 경로를 빠르게 이해하려는 사용자

## 핵심 요구사항

1. [P0] 원본 위치 좌표는 DB에서 삭제하지 않는다.
2. [P0] 지도 렌더링에 사용하는 좌표만 별도로 정리한다.
3. [P0] 실제 이동 경로에서 도로/동선과 무관하게 길게 튄 선이 없어야 한다.
4. [P0] GPS 튐 판단은 비현실적인 속도와 앞뒤 좌표 흐름에서 혼자 튀었다가 돌아오는 패턴을 우선 기준으로 한다.
5. [P1] 정상 이동 중 찍힌 좌표는 최대한 유지한다.
6. [P1] 같은 장소 주변의 잔떨림은 한 덩어리로 부드럽게 묶는다.
7. [P1] 머문 곳 주변의 GPS 흔들림은 경로선으로 보여주지 않고 중심 핀/장소 표시로 대체한다.
8. [P1] 오늘 탭, 오늘 기록, 하루 상세 지도에 동일한 경로 정리 기준을 적용한다.
9. [P2] 제외된 GPS 튐 좌표는 사용자 화면에 표시하지 않는다.

## 제약 조건 & 전제

- 원본 `location_points` 데이터는 보존한다.
- 정리 로직은 표시용 경로 생성 단계에만 적용한다.
- 단순화를 과하게 적용하면 안 된다. 핵심은 좌표 수 축소가 아니라 이상치 제거다.
- accuracy 값은 보조 기준으로만 사용한다. accuracy가 좋아도 튄 좌표가 있을 수 있다.
- 화면마다 경로 기준이 다르면 사용자가 헷갈리므로 동일한 로직을 공유해야 한다.

## 엣지 케이스 & 오류 시나리오

- 실내/도심에서 한 장소 주변 좌표가 계속 흔들리는 경우
- 이동 중 한 점만 멀리 튀었다가 다음 좌표에서 원래 동선으로 돌아오는 경우
- 실제 차량 이동처럼 빠른 이동이 정상인 경우
- 좌표가 1개뿐이라 이동 경로는 없지만 위치 지도는 보여줘야 하는 경우
- 정상 이동 좌표가 과하게 제거되어 경로가 지나치게 단순해지는 경우

## 범위 밖

- 원본 DB 좌표 삭제
- 사용자 화면에 제외 좌표를 디버그 정보로 표시
- 좌표 수를 강제로 20~40개 수준으로 압축하는 강한 단순화
- 방문한 곳 저장 UX 변경
- 알림/돌아보기 문장 생성 변경

## 수용 기준

- [ ] 지도에 도로/동선과 무관하게 길게 삐져나가는 선이 없어야 한다.
- [ ] 정상 이동 중 찍힌 좌표가 대부분 유지되어 경로가 과하게 단순해지지 않아야 한다.
- [ ] 한 장소 안의 잔떨림은 경로선 대신 중심 위치 중심으로 부드럽게 정리되어야 한다.
- [ ] 오늘 탭 상단 지도, 오늘 기록 지도, 하루 상세 지도에서 같은 데이터는 같은 형태의 경로로 보여야 한다.
- [ ] 원본 `location_points` 개수는 표시용 정리 후에도 줄어들지 않아야 한다.
- [ ] 튄 좌표를 제거해도 앱이 `LatLng is not finite` 같은 지도 렌더링 오류를 내지 않아야 한다.

## 구현 계획으로 넘길 메모

- 현재 표시 경로 생성의 중심은 `RouteDisplayPointCleaner`와 `LocationPostProcessor.cleanRouteDisplayPoints` 쪽이다.
- 다음 계획 단계에서는 현재 경로 정리 로직의 입력/출력과 각 화면의 사용 경로를 먼저 확인해야 한다.
- 속도 기반 이상치 제거와 앞뒤 흐름 기반 spike 제거는 테스트 케이스를 먼저 고정해야 한다.
